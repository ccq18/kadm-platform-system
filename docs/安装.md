# KADM 安装方式

这份文档记录当前推荐安装路径。`local/` 目录是个人备用目录，不属于安装文档范围。

## 1. 首台服务器安装

前提：

- 一台空服务器
- 服务器发行版要求：Ubuntu >= 20.04、Debian >= 10、RHEL >= 8.6、CentOS >= 8.6
- 本机可以 `ssh root@<public-ip>`
- 服务器可以访问 GitHub，或服务器本地已经准备好 `kadm-platform-assets.tgz`
- `KADM_GITHUB_TOKEN` 可访问 GitHub Actions、仓库内容和需要的 GHCR 包
- 如果应用镜像在私有 GHCR 包中，设置 `KADM_GHCR_USERNAME` 和 `KADM_GHCR_TOKEN`；脚本会在写入集群前验证 `kadm-app-configs` 当前 image tag 可拉取

默认从 GitHub Release 下载平台离线包：

```bash
ssh root@<public-ip>

export KADM_GITHUB_TOKEN=<github-token>
export KADM_GHCR_USERNAME=<github-user>
export KADM_GHCR_TOKEN=<github-token-or-package-token>

curl -fsSL https://raw.githubusercontent.com/ccq18/kadm-platform-system/main/bootstrap/install-kadm.sh | \
  bash -s -- all \
    --cluster kadm-test \
    --access-host root@<public-ip> \
    --private-ip <private-ip> \
    --dns-upstream 1.1.1.1 \
    --dns-upstream 8.8.8.8
```

默认平台离线包：

```text
https://github.com/ccq18/kadm-platform-assets/releases/download/bundle-latest/kadm-platform-assets.tgz
```

默认 system package：

```text
https://github.com/ccq18/kadm-platform-system/releases/download/system-latest/kadm-platform-system.tgz
```

如果本地已经有资源包，资源包必须位于执行安装脚本的服务器上：

```bash
ssh root@<public-ip>

export KADM_ASSET_BUNDLE_URL=file:///opt/kadm/kadm-platform-assets.tgz
export KADM_GITHUB_TOKEN=<github-token>

curl -fsSL https://raw.githubusercontent.com/ccq18/kadm-platform-system/main/bootstrap/install-kadm.sh | \
  bash -s -- all \
    --cluster kadm-test \
    --access-host root@<public-ip> \
    --private-ip <private-ip>
```

参数含义：

| 参数 | 含义 |
| --- | --- |
| `--cluster` | 本地保存的集群名 |
| `--access-host` | 客户端后续通过 SSH 访问的第一台服务器 |
| `--private-ip` | 第一台服务器的内网 IP |
| `--dns-upstream` | K3s/CoreDNS 使用的上游 DNS，可重复传入 |

如果目标机器上已有旧 K3s、CNI 或 KADM 状态，先使用脚本清理所有节点：

```bash
~/.local/bin/kadmctl reset-cluster \
  --node root@<master-public-ip> \
  --node root@<worker-1-public-ip> \
  --node root@<worker-2-public-ip> \
  --apply
```

`reset-cluster` 会逐台调用远端清理脚本，删除 K3s 服务、残留 CNI、kubelet、rancher/k3s 和 KADM 状态。

安装阶段：

| 阶段 | 行为 |
| --- | --- |
| `prepare` | 下载或读取平台离线包，再下载独立的 `kadm-platform-system` 包，导入本地缓存，安装本地工具 |
| `deploy` | 使用本地缓存安装 K3s 和平台基础组件，基础集群就绪后单独下载 `kadm-app-configs`，再配置 Release Console 和 GitOps 交付 |
| `all` | 先执行 `prepare`，再执行 `deploy` |

`prepare` 只依赖平台离线包和 `kadm-platform-system` 自己的 package。`deploy/all` 仍需要 `KADM_GITHUB_TOKEN`，因为发布控制台需要触发 GitHub Actions、下载 `kadm-app-configs`、读取 GitHub 仓库和拉取私有 GHCR 镜像。

默认交付入口使用 Gateway API / Cilium Gateway。若当前环境仍依赖 K3s Traefik / ServiceLB，对外入口可以显式使用 Traefik 模式：

```bash
~/.local/bin/kadmctl configure-delivery kadm-test \
  --app-configs-dir /path/to/kadm-app-configs \
  --ingress-mode traefik \
  --apply
```

Traefik 模式会应用 `kadm-app-configs/apps/<app>/base/ingress.yaml`，同时保留 Argo CD 对 HTTPRoute 的健康兼容配置，避免未启用 Gateway 时应用卡在不健康状态。

## 2. 本地客户端初始化

在本机执行：

```bash
curl -fsSL https://raw.githubusercontent.com/ccq18/kadm-platform-system/main/bootstrap/install-kadm-client.sh | \
  bash -s -- \
    --cluster kadm-test \
    --server root@<public-ip>
```

脚本会写入：

```text
~/.kadm/clusters/kadm-test/cluster.env
~/.kube/kadm/kadm-test.yaml
~/.local/bin/kadmctl
```

## 3. 打开 Web Console

```bash
~/.local/bin/kadmctl connect kadm-test
```

然后打开：

```text
http://127.0.0.1:18080
```

这个命令会保持 SSH tunnel 和 `kubectl port-forward` 前台运行。使用控制台时不要关闭它。

## 4. 应用发布

当前示例应用：

- `demo-hello`
- `demo-hello-spring`

应用源码仓库只负责构建并推送镜像到 GHCR。生产部署配置由 `kadm-app-configs` 维护：

```text
kadm-app-configs/apps/demo-hello/overlays/prod
kadm-app-configs/apps/demo-hello-spring/overlays/prod
```

Release Console 发布时会：

1. 触发应用仓库 GitHub Actions 构建镜像。
2. 等待工作流成功完成。
3. 更新 `kadm-app-configs` 中对应应用的 image tag。
4. 触发 Argo CD 同步。
5. 等待 Argo Rollouts 进入可切换状态。
6. 由操作员在控制台中手动 promote。

平台离线包只包含平台组件运行所需镜像，不包含业务应用镜像。

## 5. 常用命令

查看状态：

```bash
~/.local/bin/kadmctl status kadm-test
```

清理单节点后重装：

```bash
~/.local/bin/kadmctl reset-node root@<public-ip> --apply
```

清理整个集群后重装：

```bash
~/.local/bin/kadmctl reset-cluster \
  --node root@<master-public-ip> \
  --node root@<worker-1-public-ip> \
  --node root@<worker-2-public-ip> \
  --apply
```

初始化 demo 数据库等依赖：

```bash
~/.local/bin/kadmctl configure-demo-apps kadm-test \
  --app-configs-dir /path/to/kadm-app-configs \
  --db-ssh-target root@<db-host> \
  --apply
```

`configure-demo-apps` 会写入 demo Secret、初始化 MySQL 用户/数据库、重启应用 Pod，并优先等待 Argo Rollouts `Healthy`；没有 Rollout 资源时回退到 Pod readiness 检查。

# KADM Installation

This document describes the current recommended installation flow. `local/` is a private backup directory and is outside the installation documentation scope.

## 1. First Server Install

Prerequisites:

- One empty server
- Supported server distributions: Ubuntu >= 20.04, Debian >= 10, RHEL >= 8.6, CentOS >= 8.6
- SSH access such as `ssh root@<public-ip>`
- Either GitHub access from the server, or a local `kadm-platform-assets.tgz` already copied to the server
- `KADM_GITHUB_TOKEN` with access to GitHub Actions, repository content, and required GHCR packages
- If application images are private GHCR packages, set `KADM_GHCR_USERNAME` and `KADM_GHCR_TOKEN`; the script verifies the current image tags from `kadm-app-configs` before writing cluster resources

Default install from the GitHub Release bundle:

```bash
ssh root@<public-ip>

export KADM_GITHUB_TOKEN=<github-token>
export KADM_GHCR_USERNAME=<github-user>
export KADM_GHCR_TOKEN=<github-token-or-package-token>

curl -fsSL https://raw.githubusercontent.com/ccq18/kadm-platform-system/main/bootstrap/install-kadm.sh | \
  bash -s -- all \
    --cluster kadm-test \
    --access-host root@<public-ip> \
    --private-ip <private-ip> \
    --dns-upstream 1.1.1.1 \
    --dns-upstream 8.8.8.8
```

Default platform bundle:

```text
https://github.com/ccq18/kadm-platform-assets/releases/download/bundle-latest/kadm-platform-assets.tgz
```

Default system package:

```text
https://github.com/ccq18/kadm-platform-system/releases/download/system-latest/kadm-platform-system.tgz
```

If the bundle already exists locally, it must exist on the server that runs the installer:

```bash
ssh root@<public-ip>

export KADM_ASSET_BUNDLE_URL=file:///opt/kadm/kadm-platform-assets.tgz
export KADM_GITHUB_TOKEN=<github-token>

curl -fsSL https://raw.githubusercontent.com/ccq18/kadm-platform-system/main/bootstrap/install-kadm.sh | \
  bash -s -- all \
    --cluster kadm-test \
    --access-host root@<public-ip> \
    --private-ip <private-ip>
```

Install phases:

| Phase | Behavior |
| --- | --- |
| `prepare` | Downloads or reads the platform bundle, then downloads the standalone `kadm-platform-system` package, imports local cache, and installs local tools |
| `deploy` | Uses local cache to install K3s and base platform components, then fetches `kadm-app-configs` separately and configures Release Console and GitOps delivery |
| `all` | Runs `prepare`, then `deploy` |

`prepare` depends on the platform bundle plus the separate `kadm-platform-system` package. `deploy/all` still require `KADM_GITHUB_TOKEN` because Release Console triggers GitHub Actions, downloads `kadm-app-configs`, reads GitHub repositories, and pulls private GHCR images when needed.

If the target machines already contain old K3s, CNI, or KADM state, reset all nodes with the script first:

```bash
~/.local/bin/kadmctl reset-cluster \
  --node root@<master-public-ip> \
  --node root@<worker-1-public-ip> \
  --node root@<worker-2-public-ip> \
  --apply
```

`reset-cluster` runs the remote cleanup script on every node and removes K3s services, residual CNI, kubelet, rancher/k3s, and KADM state.

The default delivery ingress is Gateway API / Cilium Gateway. If the current environment still uses K3s Traefik / ServiceLB for public HTTP, configure delivery with Traefik mode:

```bash
~/.local/bin/kadmctl configure-delivery kadm-test \
  --app-configs-dir /path/to/kadm-app-configs \
  --ingress-mode traefik \
  --apply
```

Traefik mode applies `kadm-app-configs/apps/<app>/base/ingress.yaml` and keeps Argo CD HTTPRoute health compatibility so applications do not stay unhealthy when Gateway is not used.

## 4. Application Release

Current demo applications:

- `demo-hello`
- `demo-hello-spring`

Application source repositories only build and push images to GHCR. Production deployment configuration is owned by `kadm-app-configs`:

```text
kadm-app-configs/apps/demo-hello/overlays/prod
kadm-app-configs/apps/demo-hello-spring/overlays/prod
```

Release Console:

1. Triggers the application GitHub Actions image build.
2. Waits for the workflow to finish successfully.
3. Updates the matching image tag in `kadm-app-configs`.
4. Triggers Argo CD sync.
5. Waits for Argo Rollouts to reach the switch point.
6. Leaves promote as an explicit operator action.

The platform offline bundle includes only platform runtime images, not business application images.

## 5. Common Commands

Check status:

```bash
~/.local/bin/kadmctl status kadm-test
```

Reset one node before reinstalling:

```bash
~/.local/bin/kadmctl reset-node root@<public-ip> --apply
```

Reset the whole cluster before reinstalling:

```bash
~/.local/bin/kadmctl reset-cluster \
  --node root@<master-public-ip> \
  --node root@<worker-1-public-ip> \
  --node root@<worker-2-public-ip> \
  --apply
```

Initialize demo database dependencies:

```bash
~/.local/bin/kadmctl configure-demo-apps kadm-test \
  --app-configs-dir /path/to/kadm-app-configs \
  --db-ssh-target root@<db-host> \
  --apply
```

`configure-demo-apps` writes demo Secrets, initializes MySQL users/databases, restarts app Pods, and waits for Argo Rollouts `Healthy`; if no Rollout resource exists it falls back to Pod readiness.

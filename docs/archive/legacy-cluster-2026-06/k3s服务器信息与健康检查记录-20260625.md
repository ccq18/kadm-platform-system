## 4. K3s Server 配置摘要

`/etc/rancher/k3s/config.yaml` 当前关键配置如下：

```yaml
node-name: "k3s-master-1"
node-ip: "10.120.0.2"
node-external-ip: "157.230.140.91"
flannel-iface: "eth1"
write-kubeconfig-mode: "600"
egress-selector-mode: "pod"
disable-agent: true
tls-san:
  - "157.230.140.91"
  - "10.120.0.2"
kubelet-arg:
  - "resolv-conf=/etc/rancher/k3s/resolv.conf"
```

配置含义：

- `disable-agent: true`：master 不承担 worker 角色，所以 `kubectl get nodes` 里只看到两个 worker。
- `node-ip` 使用私网：节点间组件通信走 `10.120.0.x`。
- `flannel-iface: eth1`：Pod 网络优先走私网网卡。
- `kubelet resolv-conf`：Pod DNS 使用自定义 resolv 配置，避免重复 nameserver 告警。
- `egress-selector-mode: "pod"`：让 agentless master 也能正常访问集群内 Pod / Service。

建议：

- 如果要承载持久化业务，先明确卷目录、备份方式和恢复流程。

### 8.4 本次部署中已发现并修复的实际问题

现象：

- 部署后最初 `kubectl top nodes` 返回 `Metrics API not available`

根因：

- master 采用 `disable-agent: true`
- 未配置 `egress-selector-mode`
- 导致 apiserver 不能正常访问集群内部 `metrics-server`

修复：

- 在 master 配置中加入：

```yaml
egress-selector-mode: "pod"
```

- 重启 `k3s`
- 重启两个 worker 的 `k3s-agent`

结果：

- `kubectl top nodes`
- `kubectl top pods -A`

现已验证可正常返回。
